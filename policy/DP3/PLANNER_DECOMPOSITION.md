# DeepSeek task decomposition workflow

This workflow converts RoboTwin task instructions into structured planner supervision with the DeepSeek API and then converts those task-level decompositions into per-timestep planner labels.

## Step 1: generate task decompositions

### Input format

The CLI expects a JSON file with two top-level lists:

```json
{
  "seen": ["instruction 1", "instruction 2"],
  "unseen": ["instruction 3"]
}
```

### Environment

Set a DeepSeek API key before running the script:

```bash
export DEEPSEEK_API_KEY=your_key_here
```

### Run decomposition

```bash
python policy/DP3/scripts/decompose_tasks.py \
  policy/DP3/examples/task_decomposition_input.json \
  --output policy/DP3/examples/task_decomposition_output.json \
  --jsonl-output policy/DP3/examples/task_decomposition_output.jsonl \
  --deduplicate
```

### Decomposition output shape

Each instruction is expanded into a normalized planner record with:

- `instruction`
- `canonical_task`
- `task_id`
- `task_category`
- `scene_objects`
- `final_goal`
- `stages[]`

Each `stages[]` entry contains:

- `stage`
- `stage_type`
- `action_type`
- `source_object`
- `target_object`
- `support_object`
- `target_support`
- `target_region`
- `target_location`
- `spatial_relation`
- `preferred_arm`
- `required_objects`
- `completed_subgoals_before_stage`
- `success_criteria`

## Step 2: build per-timestep planner labels

Use `build_planner_labels.py` to turn the decomposition output into `planner_labels.jsonl`.

### Required inputs

You need:

1. A decomposition file generated by `decompose_tasks.py` (`.json` or `.jsonl`)
2. A mapping from `episode_id` to the exact instruction used in that episode
3. Episode transition lengths

### Minimal example

```bash
python policy/DP3/scripts/build_planner_labels.py \
  --decomposition-file policy/DP3/examples/task_decomposition_output.json \
  --episode-instruction-map policy/DP3/examples/episode_instruction_map.json \
  --episode-lengths-json policy/DP3/examples/episode_lengths.json \
  --output policy/DP3/examples/planner_labels.jsonl \
  --summary-output policy/DP3/examples/planner_labels_summary.json
```

### Alternative input mode

If you already have per-episode instruction files like `episode0.json`, `episode1.json`, ... you can use:

```bash
python policy/DP3/scripts/build_planner_labels.py \
  --decomposition-file policy/DP3/examples/task_decomposition_output.json \
  --instruction-dir /path/to/instructions \
  --instruction-source seen_first \
  --episode-lengths-json policy/DP3/examples/episode_lengths.json \
  --output policy/DP3/examples/planner_labels.jsonl
```

### Alignment mode

By default the script uses a **uniform partition** baseline: it splits each episode evenly across all planner stages.

By default the script also applies **stack-stage normalization** before alignment. For stack tasks, this collapses redundant `move/place_*_at_center` stages into a more execution-oriented sequence such as `place_red_at_center -> stack_green_on_red -> stack_blue_on_green`. If you want to keep the raw decomposition stages, add:

```bash
--disable-stage-normalization
```

If you already have explicit stage boundaries, pass a JSON file like:

```json
{
  "0": [
    {"start": 0, "end": 2},
    {"start": 2, "end": 4},
    {"start": 4, "end": 6}
  ]
}
```

with:

```bash
--stage-boundaries-json your_boundaries.json
```

### Planner label output shape

Each output JSONL row contains:

- `flat_index`
- `episode_id`
- `timestep`
- `instruction`
- `split`
- `canonical_task`
- `task_category`
- `final_goal`
- `stage`
- `stage_type`
- `stage_index`
- `num_stages`
- `is_terminal_stage`
- `source_object`
- `target_object`
- `support_object`
- `target_region`
- `target_support`
- `target_location`
- `spatial_relation`
- `preferred_arm`
- `required_objects`
- `completed_subgoals`
- `relevance_objects`
- `success_criteria`

This output is meant for planner supervision. DP3 should continue reading the original zarr with `point_cloud`, `state`, and `action`.


## Step 3: use planner labels for training

A first training hook is now available in `diffusion_policy_3d/dataset/planner_robot_dataset.py`.
It loads the original DP3 zarr plus `planner_labels.jsonl`, aligns them by `flat_index`, and returns a `planner` dict containing integer ids for stage and object supervision.

This is intended as the data interface for planner-conditioned training; model and loss integration are still up to the training code.
